## 1. 产品概述

### 1.1 产品定位

SnapSport C端客户端是面向南京本地高校、社区和业余运动用户的约球订场与运动数据服务入口。产品以「组局、订场、AA结算」作为高频刚需切入，逐步叠加 AI 高光集锦、运动数据报告、球员能力评分、球队管理和数据贡献激励体系，形成从运动前、运动中到运动后的完整闭环。

### 1.2 核心目标

1. 在 MVP 阶段解决用户「找不到人、订不到场、费用难分摊、爽约难治理」的问题。
2. 在增长阶段通过 AI 高光集锦和运动数据报告提升用户分享率、复购率和平台差异化。
3. 在数据闭环阶段建立用户主动贡献运动数据的机制，为后端运动数据资产沉淀提供稳定来源。

### 1.3 目标用户

| 用户类型 | 典型场景 | 核心诉求 |
| --- | --- | --- |
| 高校学生 | 课后篮球、足球、羽毛球约局 | 快速组队、低价订场、找到同水平队友 |
| 社区运动用户 | 晚间或周末固定运动 | 稳定球局、减少爽约、便捷 AA |
| 球队队长 / 群主 | 组织训练、比赛、联赛报名 | 队员管理、通知、出勤、费用结算 |
| 数据贡献用户 | 使用运动相机、可穿戴设备记录运动 | 获得运动币、现金奖励和专属权益 |

## 3. 用户核心流程

### 3.1 约球流程

1. 用户登录并完善个人资料、运动偏好、位置和水平等级。
2. 用户进入首页，按运动类型、位置、时间、水平筛选球局。
3. 用户选择加入公开局，或通过邀请链接加入熟人局。
4. 系统判断用户信用分、人数限制和支付状态。
5. 用户完成预付占位。
6. 球局满员后自动锁局，并通知全员。
7. 开局前 2 小时推送提醒。
8. 到场后完成核销或签到。
9. 球局结束后进入费用结算、互评和信用分更新。

### 3.2 发起组局流程

1. 发起人选择运动类型、场馆、时间、人数、水平要求和局类型。
2. 发起人填写预计费用，包括场地费、球费和其他费用。
3. 系统自动计算人均预付金额。
4. 发起人选择公开局或熟人局。
5. 系统生成球局详情页和邀请链接。
6. 报名人数满员后自动锁局。
7. 如有人临时退出，系统触发补人机制。

### 3.3 场地预订流程

1. 用户选择运动类型、日期、区域和价格区间。
2. 系统展示可预订场馆与时段库存。
3. 用户选择场地和时段。
4. 用户直接预订，或绑定到某个球局。
5. 用户完成支付。
6. 系统生成订单和核销码。
7. 场馆端核销后，订单状态更新。

### 3.4 赛后互评流程

1. 球局结束后自动开启 24 小时互评通道。
2. 同场用户可对队友进行匿名评分。
3. 单场至少收到 3 条有效评分后，评分结果生效。
4. 系统按动态权重更新用户能力分与等级。
5. 组队页展示用户综合分、场均分和能力标签。

## 4. 功能需求

## 4.1 模块一：基础约球与交易系统

### 4.1.1 智能组局

**功能目标**  
帮助用户快速发起或加入匹配的运动局，提升组队效率和到场率。

**功能说明**

| 功能点 | 需求描述 | 优先级 |
| --- | --- | --- |
| 发起球局 | 支持篮球、足球、羽毛球三类运动，填写时间、地点、人数、费用、水平要求 | P0 |
| 加入球局 | 用户可按运动类型、距离、时间、水平筛选并加入公开局 | P0 |
| 熟人局 | 仅通过邀请链接或指定用户可见 | P1 |
| 公开局 | 进入平台列表与推荐流，可被附近用户搜索和报名 | P0 |
| 人数展示 | 实时展示已报名人数、剩余名额、参与者头像和基础标签 | P0 |
| 满员锁局 | 达到人数上限后自动停止报名，并通知所有参与者 | P0 |
| 临时补人 | 有人退出后重新开放缺口，向候补用户和附近用户推送 | P1 |
| 换人申请 | 已报名用户可申请由其他用户替换，发起人确认后生效 | P2 |
| 开局提醒 | 开局前 2 小时自动推送提醒 | P0 |

**关键规则**

- 信用分低于 80 分的用户不可加入公开局。
- 信用分低于 60 分的用户不可发起或加入任何球局。
- 已满员球局进入锁定状态，普通用户不可继续报名。
- 熟人局不进入公开推荐列表。

### 4.1.2 AA 收款与费用结算

**功能目标**  
解决运动费用分摊、预付占位和场馆结算问题，减少线下转账和逃单。

**功能说明**

| 功能点 | 需求描述 | 优先级 |
| --- | --- | --- |
| 费用预设 | 发起组局时填写场地费、球费、其他费用 | P0 |
| 自动平摊 | 系统按参与人数自动计算人均金额 | P0 |
| 在线预付 | 支持微信 / 支付宝支付，MVP 可先接模拟支付或微信支付 | P0 |
| 支付占位 | 用户支付成功后报名才正式生效 | P0 |
| 额外消费 | 局内可追加水、装备租赁等费用，并再次平摊补收 | P2 |
| 退款处理 | 用户在允许时间内取消，按规则退还费用 | P1 |
| 爽约扣费 | 开场前 1 小时内取消或无故缺席，费用不退或部分扣除 | P1 |
| 场馆结算 | 到场核销后，系统向场馆生成待结算金额 | P0 |

**结算规则**

- 报名必须完成预付，否则仅保留为未支付状态。
- 球局未成局时，系统自动触发退款。
- 用户爽约时，平台可按规则扣除费用，用于补偿场馆或其他用户。
- 场馆核销后，订单进入可结算状态。

### 4.1.3 爽约信用分体系

**功能目标**  
通过信用分约束用户行为，提升到场率和平台交易可信度。

**初始规则**

| 行为 | 信用分变化 |
| --- | --- |
| 新用户注册 | 初始 100 分 |
| 开局前 24 小时内取消 | -5 分 |
| 开场前 1 小时内取消 | -20 分 |
| 无故缺席 | -20 分 |
| 按时到场 | +1 分 |
| 成功组织球局 | +2 分 |
| 获得队友好评 | +1 分 |

**限制规则**

| 信用分区间 | 权限限制 |
| --- | --- |
| 80 - 100 | 正常使用 |
| 60 - 79 | 不可加入公开局，可加入熟人局 |
| 0 - 59 | 不可发起或加入球局 |

**产品要求**

- 个人中心展示当前信用分、变化记录和限制说明。
- 球局详情页在报名时校验信用分。
- 信用事件永久记录，支持运营后台查询。

### 4.1.4 场地预订与赛事报名

**功能目标**  
接入南京合作场馆资源，支持用户按时段预订和报名赛事。

**场地预订功能**

| 功能点 | 需求描述 | 优先级 |
| --- | --- | --- |
| 场馆列表 | 展示南京合作场馆，支持区域、运动类型、价格筛选 | P0 |
| 时段库存 | 展示日期、时段、剩余库存和价格 | P0 |
| 场地详情 | 展示地址、设施、营业时间、导航、联系方式 | P0 |
| 直接预订 | 用户可独立预订场地，不绑定球局 | P0 |
| 绑定球局 | 发起组局时可选择已合作场馆时段 | P0 |
| 核销码 | 支付成功后生成订单核销码 | P0 |

**赛事报名功能**

| 功能点 | 需求描述 | 优先级 |
| --- | --- | --- |
| 赛事列表 | 展示社区联赛、高校联赛和平台活动 | P1 |
| 单人报名 | 用户可单独报名，后续由平台组队或匹配 | P1 |
| 球队报名 | 队长可代表球队报名赛事 | P1 |
| 赛程生成 | 报名后展示赛程、对手、场地和时间 | P2 |
| 赛前提醒 | 比赛前推送提醒和导航信息 | P1 |

### 4.1.5 球员运动能力评分系统

**功能目标**  
建立用户运动水平标签，提升组局匹配质量，减少实力差距导致的体验问题。

**评分维度**

建议 MVP 使用 5 个维度：

| 维度 | 说明 |
| --- | --- |
| 体能 | 耐力、速度、对抗强度 |
| 技术 | 控球、投篮 / 射门、传接球能力 |
| 意识 | 跑位、配合、攻防判断 |
| 稳定性 | 发挥稳定程度、失误控制 |
| 协作 | 团队沟通、守规则、配合意愿 |

**自评规则**

- 个人中心开放 5 维度自评入口。
- 提供 5 档快捷等级一键填充：新手、入门、普通、熟练、高手。
- 用户提交后 7 天内最多修改 1 次。
- 自评分作为冷启动参考，权重随互评数据增加逐步降低。

**互评规则**

- 球局结束后开启 24 小时互评通道。
- 支持对同场用户批量匿名打分。
- 单场至少 3 条有效评分后，该场互评结果才计入能力分。
- 用户不可评价自己。
- 明显异常评分进入风控或降权处理。

**综合分计算**

综合分由自评分、赛后互评、到场履约和历史稳定性构成。

| 数据来源 | 冷启动权重 | 稳定期权重 |
| --- | --- | --- |
| 个人自评 | 60% | 15% |
| 同场互评 | 30% | 55% |
| 履约与信用 | 10% | 15% |
| 历史表现稳定性 | 0% | 15% |

**等级映射**

| 综合分 | 等级 |
| --- | --- |
| 0 - 20 | 新手 |
| 21 - 40 | 入门 |
| 41 - 60 | 普通 |
| 61 - 80 | 熟练 |
| 81 - 100 | 高手 |

**展示位置**

- 个人中心展示个人综合分、5 维雷达图、历史变化趋势。
- 球局详情页展示参与者等级、综合分和场均分。
- 发起组局时可设置推荐水平区间。
- 推荐系统基于运动类型、位置、时间和能力分匹配球局。

## 4.2 模块二：AI 运动数据服务

### 4.2.1 自动高光集锦生成

**功能说明**

| 功能点 | 需求描述 | 优先级 |
| --- | --- | --- |
| 扫码绑定 | 用户到场后扫描场地二维码，绑定账号与场次 | P1 |

**前置依赖**

- 场馆摄像头或上传视频源。
- 场次与用户身份绑定。

### 4.2.2 个人运动数据报告

## 4.3 模块三：运动数据贡献与激励体系

### 4.3.1 数据上传入口

**功能目标**  
让用户自愿上传运动数据，并明确授权范围和隐私边界。

**数据来源**

| 来源 | 说明 |
| --- | --- |
| 场馆固定摄像头 | 第三人称运动视频和轨迹数据 |
| 用户自有设备 | 联名运动相机、智能手环、智能手表等第一视角或传感器数据 |

**授权选项**

- 上传范围：全场数据 / 个人片段。
- 使用范围：仅用于模型训练 / 可公开脱敏使用。
- 隐私处理：上传前或入库前自动脱敏，去除人脸、声音等个人隐私信息。

### 4.3.2 Play-to-Earn 激励机制

**功能目标**  
通过运动币、现金奖励和长期权益，激励用户贡献高质量运动数据。

**质量评分维度**

| 维度 | 说明 |
| --- | --- |
| 画面清晰度 | 是否模糊、遮挡、过暗 |
| 动作完整度 | 动作是否完整可识别 |
| 场景丰富度 | 是否包含对抗、转身、急停、变向等高价值场景 |
| 数据时长 | 有效运动数据持续时间 |

**奖励类型**

| 奖励 | 说明 |
| --- | --- |
| 基础奖励 | 有效数据兑换平台运动币，可抵扣场地费或兑换周边 |
| 高额奖励 | 高质量专项数据可获得现金奖励 |
| 长期权益 | 会员、联名硬件优先购买权、线下赛事免费名额 |

## 5. 非功能需求

### 5.1 性能要求

- 首页球局列表首屏加载时间小于 2 秒。
- 支付结果回调后订单状态更新小于 3 秒。
- 球局报名人数变化应尽量实时刷新。
- AI 集锦生成完成后 10 分钟内通知用户。

### 5.2 安全与隐私

- 用户手机号、支付信息、定位信息需加密存储或脱敏展示。
- 运动视频和数据上传必须经过用户授权。
- 默认不公开用户原始视频。
- 数据商业化使用必须遵循用户授权范围。
- 未成年人用户应在后续版本补充监护与隐私保护机制。

### 5.3 风控要求

- 防止虚假报名、恶意取消、刷评分、刷奖励。
- 互评结果需支持异常检测与人工复核。
- 数据上传奖励需经过质量校验后发放。

## 8. 后台与跨端依赖

### 8.1 C端依赖后台能力

| 后台能力 | 说明 |
| --- | --- |
| 用户管理 | 用户资料、信用分、封禁、角色 |
| 场馆管理 | 场馆资料、场地、时段、价格、库存 |
| 订单管理 | 支付、退款、核销、对账 |
| 球局管理 | 创建、报名、取消、锁局、补人 |
| 赛事管理 | 赛事、赛程、报名、榜单 |
| 内容管理 | 动态、评论、举报、活动 |
| 数据管理 | 视频、报告、数据授权、奖励 |

### 8.2 B端场馆系统对接

- C端预订订单需同步至场馆端。
- 场馆端可查看订单、核销订单、管理时段库存。
- 场馆摄像头采集视频后，根据扫码绑定结果推送到对应用户账号。

### 8.3 硬件设备对接

- 支持联名运动相机绑定。
- 支持智能手环、手表等可穿戴数据导入。
- 后端需统一不同设备的数据格式。

### 8.4 数据资产后台对接

- 用户上传数据经过脱敏和质量评分后进入数据资产库。
- 奖励账户根据质量评分和任务规则更新。
- 商业化使用必须校验用户授权范围。

## 9. 验收标准

### 9.1 MVP 验收标准

1. 用户可以完成注册、登录、资料编辑。
2. 用户可以创建球局并设置运动类型、时间、地点、人数、费用和水平要求。
3. 用户可以加入公开局并完成支付占位。
4. 球局满员后可以自动锁局。
5. 用户可以直接预订场地并生成订单。
6. 场馆端可以核销订单。
7. 用户取消、缺席、按时到场后，信用分按规则变化。
8. 用户可以完成 5 维度自评。
9. 球局结束后可以进行匿名互评，并在满足条件后更新能力分。
10. 队长可以创建球队并管理成员。

### 9.2 AI 阶段验收标准

1. 用户扫码后可以绑定个人账号与场次。
2. 场次结束后用户可收到高光集锦。
3. 用户可以下载和分享集锦。
4. 用户可以查看基础运动数据报告。

### 9.3 数据闭环阶段验收标准

1. 用户可以上传指定范围的运动数据。
2. 用户可以选择授权范围。
3. 系统可以展示数据质量评分。
4. 用户可以获得运动币或现金奖励记录。
5. 贡献者榜单和任务大厅可正常展示。

## 10. 风险与待确认问题

| 问题 | 风险 | 建议 |
| --- | --- | --- |
| 支付与退款复杂度较高 | 影响 MVP 上线周期 | MVP 可先做模拟支付或单一微信支付 |
| 场馆库存实时性不足 | 可能出现重复预订 | 先与 3-5 家场馆建立人工确认机制，再逐步系统化 |
| AI 高光依赖硬件和算法 | 增长期交付风险较高 | MVP 先做 Demo 入口和视频上传样例 |
| 互评可能被刷分 | 影响匹配公平 | 引入匿名、最低评分数、异常评分降权 |
| 数据上传涉及隐私 | 合规风险较高 | 明确授权、脱敏、用途限制和撤回机制 |
